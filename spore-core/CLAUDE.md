# spore-core — Spore VM

## What is this?

A tiny, stack-based language VM designed to be **generated by AI, not written by humans**. Inspired by Forth. The VM is `#![no_std]` with zero dependencies and zero allocations — everything uses fixed-capacity const-generic data structures.

A spore is a program — a tiny capsule of instructions that lands on a microcontroller and starts doing things.

### Why "Spore"?

- **Biological** — a self-contained capsule of instructions that germinates on contact with a host. That's literally what flashing a program to an ESP32 is.
- **Scale** — spores are microscopic. This VM is <8KB. The name *sounds* tiny.
- **Our world** — cacti, aeroponics, cell simulation. Spore is native to that vocabulary. "Deploy a spore to the greenhouse node" just works.
- **AI generation** — spores are produced and dispersed, not hand-crafted. That's the AI-first design philosophy in one word.
- **Not derivative** — it doesn't set up expectations from another project. It stands on its own.

## Architecture

```
Token Stream (text) → Parser → [Op] array → VM execution
                        ↓           ↓
                    StringPool   Dict (words)
                    BufferPool
```

### Source files

| File | What |
|---|---|
| `lib.rs` | Crate root. `VmError`, `VmAction` enums, re-exports. |
| `value.rs` | `Value` enum — tagged stack cell: `I(i32)`, `F(f32)`, `B(bool)`, `S(u16)`, `Buf(u16)`. Type promotion helpers. |
| `op.rs` | `Op` enum — all opcodes (literals, stack, arithmetic, comparison, logic, type conversion, control flow, variables, words, tasks, events, platform). |
| `stack.rs` | `Stack<N>` — fixed-size const-generic stack. Returns `Result` on push/pop. |
| `strings.rs` | `StringPool<BYTES, COUNT>` — fixed-capacity string interning with deduplication. |
| `buffer.rs` | `BufferPool<BYTES, COUNT>` — fixed-capacity raw byte buffer pool for I2C/SPI. |
| `dict.rs` | `Dict<N>` — word name (string pool index) → program offset mapping. |
| `vm.rs` | `Vm` — opcode dispatch, `step()`/`run()`/`run_steps()`. Owns stacks, program, strings, buffers, platform. Pushes `VmAction`s for scheduler. |
| `parse.rs` | Token stream text → `[Op]` compiler. Resolves control flow offsets, word definitions, string/float/hex literals. |
| `task.rs` | `Task` (per-task stacks, IP, vars, EVERY/TIMES state) + `Scheduler` (round-robin, event bindings, event queue, action processing). |
| `platform.rs` | `Platform` trait — HAL abstraction. All methods default to `Err(PlatformError)`. Backends implement what they support. |

### Key types

- **`Vm<P, DS, RS, STR_BYTES, STR_COUNT, BUF_BYTES, BUF_COUNT, PROG, VARS>`** — the VM, generic over platform and all capacities.
- **`Scheduler<P, N, DS, RS, E, VARS>`** — round-robin scheduler with `N` task slots, `E` event binding slots, and `VARS` variable slots per task.
- **`VmAction`** — `StartTask(u16)`, `StopTask(u16)`, `EmitEvent(u16)`, `BindEvent { event_id, word_offset }`. VM pushes these; scheduler drains after each tick.

## Token Stream Format

All tokens uppercase, space-delimited. AI generates this; parser compiles to `Op` sequences.

```
DEF double DUP ADD END
TASK main LIT 21 double HALT ENDTASK
```

Control flow is structured (IF/ELSE/THEN, LOOP/ENDLOOP, TIMES/ENDTIMES, BEGIN/UNTIL) — no raw jumps. Forward references resolved via fixup stack during parsing.

## Design Decisions

- **No alloc.** All data structures (stacks, string pool, buffer pool, dict, program, vars, event bindings) are fixed-capacity arrays with const generic sizes.
- **`VmAction` queue.** The VM cannot directly manipulate tasks — it pushes actions that the scheduler drains. This keeps the VM decoupled from scheduling.
- **Per-task state.** EVERY/TIMES counters, stacks, vars, and IP are all per-task. The scheduler swaps them into/out of the VM on context switch.
- **Event dispatch.** `ON <event> <word>` registers a binding in the scheduler. `EMIT` or `scheduler.emit_event()` queues events. On next tick, suspended tasks with matching bindings are woken with IP set to the handler word.
- **Platform trait defaults to error.** Backends only implement what they support. Calling an unimplemented peripheral op is a clean `PlatformError`, not a panic.

## Platform Method Names (matches spec)

I2C: `i2c_set_addr`, `i2c_write_byte`, `i2c_read_byte`, `i2c_write_buf`, `i2c_read_buf`, `bme_read`
WiFi: `wifi_connect`, `wifi_status` (`&self`), `wifi_disconnect`, `wifi_ip`
System: `millis` (`&self`), `nvs_get` (`&self`), `heap_free` (`&self`) — these are `&self`, rest are `&mut self`

Return types are `PlatformResult<T>` (`Result<T, VmError>`) everywhere, even where the spec says `Option` or plain values — this is intentional for `?` ergonomics.
